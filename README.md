# PhoneCheck AI

**A computer-vision pipeline that grades second-hand smartphone condition from photos and estimates repair cost.**

Built from scratch as a first-year deep learning project: dataset collected and labeled by hand, five custom-trained models, served behind a guided mobile-friendly Streamlit app with a vision-LLM assistant.

## What it does

A user picks their phone brand and model, then walks through a guided capture wizard that takes 2–8 photos depending on scan mode (Quick, Standard, Accurate). For each photo, the app:

1. **Validates the shot** with a view-gate model — rejects wrong-side or non-phone images at capture time
2. **Classifies the surface** — front screens and back panels are graded as `cracked`, `good`, or `scratched`
3. **Inspects the edges** — a custom YOLOv11 detector crops the phone frame out of each edge photo, splits it into three overlapping tiles, and runs an edge-wear classifier on each tile (worst tile wins)
4. **Produces a verdict** — an overall grade (`mint` / `used` / `damaged`), confidence score, repair-cost estimate in EUR based on the selected model, and a plain-language explanation generated by a local vision-LLM

<p align="center">
  <img src="docs/screenshots/02_selector.png" alt="Device selector" width="640">

The system catches common gaming attempts — for example, submitting a back photo when asked for the front — and applies a conservative bias: if "mint" prediction confidence is below 60%, the second-place class wins. False alarms cost a delayed sale; false negatives cost €100–300 in unflagged repairs.

## The dataset — built from scratch

I collected and labeled the dataset myself across **~4,000 images** (including augmented variants). Real images came from:

- Second-hand phone marketplaces (eBay, Marktplaats, iFixit teardowns)
- Direct photography of devices in various conditions
- Two public Kaggle phone-damage datasets (used to supplement the scratched class only)

**Breakdown of real images (pre-augmentation):**

| Class | Front | Back | Edges |
|---|---|---|---|
| Cracked / damaged | 485 | 335 | — |
| Good / mint | 199 | 245 | 192 (merged) |
| Scratched / used | 98 (incl. Kaggle) | 69 | 157 (merged) |

Hashing the full file path (not just filename) for cross-validation group IDs caught 13 filename collisions across classes that would have caused silent data leakage. Catching this turned out to be one of the more impactful methodology improvements in the project.

## The models

Five production models, all trained with MobileNetV2 as the backbone (chosen for mobile-deployable size — ~2.4M parameters each) plus a YOLOv11n object detector. Accuracy ranges below show seed-to-seed variance on the held-out test set:

| Model | File | Architecture | Test Accuracy | Purpose |
|---|---|---|---|---|
| **View Gate** | `phone_side_classifier.keras` | MobileNetV2 + 3-class head | **92–97%** | Validates front/back/other before main classifiers run |
| **Front CNN** | `phone_classifier_front.keras` | MobileNetV2 + 3-class head + SHARPEN preprocessing | **85–89%** | cracked / good / scratched |
| **Back CNN** | `phone_classifier_back.keras` | MobileNetV2 + 3-class head + CLAHE preprocessing | **83–88%** | cracked / good / scratched |
| **Edge CNN** | `edge_classifier_merged.keras` | MobileNetV2 + 2-class head + YOLO crop + 3-tile worst-wins | **70–77%** | mint / used (edge wear) |
| **YOLO Edge Detector** | `best_yolo_v22.pt` | YOLOv11n (custom-trained on 329 annotated edges) | **99.5% mAP** | Crops phone edge from background before edge CNN |

### What I built and why it works

- **Per-model preprocessing.** A/B-tested CLAHE, SHARPEN (UnsharpMask), and raw inputs across all five models using 5-fold cross-validation. The result is per-model: SHARPEN helps the front model (scratches are fine high-frequency detail), CLAHE helps the back model (camera modules and logos provide texture context), and raw images win on edges and the view gate. No universal pipeline — each model gets the preprocessing it actually benefits from.

- **YOLO + tiling for edges.** Edge photos are long, narrow rectangles. Resizing them to 224×224 destroys the fine scratches that distinguish mint from used. I trained a YOLOv11 detector to crop the edge cleanly, then split each crop into 3 overlapping tiles along the long axis (15% overlap) and resize each tile to 224×224 independently. Inference uses "worst-tile-wins" — any tile saying "used" makes the whole edge "used". This conservative bias is intentional: it's safer to flag a mint edge as used than miss real damage.

- **Confidence threshold for "good" predictions.** If the "good" / "mint" class has below 60% softmax confidence, the second-place class wins. This trades a small drop in accuracy for a meaningful reduction in dangerous false negatives — the failure mode that actually costs money in a resale platform.

- **Merged edge model.** I originally trained separate models for side edges and top/bottom edges, but each had only ~140 and ~210 real images respectively. Merging them into a single "edge wear" model gave the model more diverse training data and produced the single largest accuracy jump of the entire project.

- **View gate.** Trained on real front/back images plus CIFAR-10 and synthetic noise images as the "other" class. Catches users submitting wrong-side photos or non-phone images before they hit the main classifiers.

Full methodology — 12 iterations from a basic CNN baseline through transfer learning, augmentation, multi-class extension, per-model preprocessing, YOLO crop + tiling, and the view gate — is documented in [`notebooks/Deep-Learning-Parham-250997.ipynb`](notebooks/Deep-Learning-Parham-250997.ipynb).

## The full inference pipeline

```
User photo
   ↓
View Gate ─── "other"? ─── reject, prompt retake
   ↓ (front / back / edge)
   │
   ├── Front → SHARPEN → Front CNN ──┐
   ├── Back  → CLAHE   → Back CNN  ──┤
   └── Edge  → YOLO crop → 3 tiles  ──┤
                                      ↓
                          Aggregate + threshold + edge override
                                      ↓
                          Vision LLM (Ollama / qwen3-vl)
                                      ↓
                Final verdict + confidence + repair-cost estimate
```

**Edge override rule:** if 2 or more edges are classified as "used", the overall grade drops to "used" regardless of front/back condition. Catches phones with mint screens but scuffed frames — common in real-world trade-ins.

## Tech stack

- **Models:** TensorFlow / Keras (MobileNetV2 transfer learning), Ultralytics YOLOv11
- **App:** Streamlit, OpenCV, Pillow
- **LLM integration:** Ollama running a local vision model (qwen3-vl:8b)
- **Object detection:** Custom YOLOv11n trained via Roboflow

## Running the app locally

**Requirements:** Python 3.10+, ~2GB RAM during inference, optionally Ollama with a vision model for the LLM features.

```bash
git clone https://github.com/<your-username>/phonecheck-ai
cd phonecheck-ai
pip install -r requirements.txt
streamlit run app/app.py
```

App opens at `http://localhost:8501`. First load takes 10–20 seconds while the 5 models cache.

### Optional: LLM assistant

For the natural-language condition descriptions and post-scan chat, install [Ollama](https://ollama.com) and pull a vision model:

```bash
ollama pull qwen3-vl:8b
```

The app degrades gracefully without Ollama — classification works as normal, just without the chat assistant.

## What I learned

- **Methodology beats architecture tweaks.** The 13 colliding filenames I found by switching from filename-hashing to path-hashing for CV group IDs had a bigger impact on real accuracy than any architecture change I tried.
- **Per-task preprocessing matters more than I expected.** Assuming one preprocessing pipeline ("just normalize and augment") fits every model is wrong. SHARPEN helps front, CLAHE helps back, raw inputs win on edges and the view gate. The A/B tests took time but paid off.
- **Tile-and-worst-wins is a cheap way to get conservative bias.** On a problem where false negatives are expensive (passing damage off as good), ensembling tiles with a worst-case aggregator is a low-effort way to bias toward safety.
- **A working app is more compelling than a notebook.** I could have stopped at the 12th iteration in the notebook — the models work. Putting them behind a guided UI with positioning guides, a results page, and a chat assistant turned the work into something a non-technical person can actually use, which made everything else easier to explain.
